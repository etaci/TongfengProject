# 痛风小助手开发路线

本项目开发过程中，全部中文编码使用UTF-8

## 一、文档目的
本文档基于项目规划书与《技术架构.md》整理，目标不是给出按时间划分的排期，而是明确项目后续开发应遵循的实现顺序、模块依赖、交付边界与推进原则。

本项目的最终目标始终是“痛风主动管理平台”，但开发路线必须遵循“先跑通核心闭环，再增强可信能力，再升级主动管理，最后扩展平台生态”的路径。

## 二、总体开发原则

### 2.1 北极星不变，落地顺序收敛
- 最终方向保持为主动管理平台
- 首次开发不追求大而全
- 所有功能必须围绕“记录、识别、提醒、干预、留存”主链路展开

### 2.2 优先做高频、低风险、强感知能力
- 优先开发用户最容易感知价值的功能
- 优先开发能够形成连续使用习惯的功能
- 对医疗风险高、责任边界模糊的能力延后上线

### 2.3 按依赖推进，不按想象力推进
- 先搭底座，再做闭环
- 先做结构化数据，再做个性化画像
- 先做规则驱动，再做预测驱动
- 先做工具价值，再做平台依赖

### 2.4 技术路线遵循既定双栈方案
- `Java + Spring Boot` 负责主业务系统
- `Python + FastAPI` 负责 AI 能力服务
- `Redis + MySQL + 对象存储 + 向量数据库` 作为基础数据设施
- 前期采用轻量编排，后期再逐步引入复杂 Agent 能力

## 三、开发顺序总览

建议按照以下顺序推进：

1. 先完成项目基础底座
2. 再完成核心记录与识别闭环
3. 再补齐数据理解与可信问答能力
4. 再建设主动预警与个性化干预能力
5. 最后扩展家属协同、硬件接入与积分生态

这条路线的核心逻辑是：
没有基础档案和记录数据，就无法做画像；
没有画像和规则体系，就无法做安全的主动预警；
没有稳定留存和连续行为，就不应该过早投入硬件和积分商城。

## 四、开发阶段路线

## 4.1 路线一：基础底座建设

### 目标
搭建后续所有业务模块共用的系统骨架，让前端、Java 主业务、Python AI 服务能够稳定协同。

### 必做内容
- 微信小程序基础工程
- 安卓应用前端架构预留
- Web 用户端前端架构预留
- Java 后端基础工程
- Python AI 服务基础工程
- API 网关或统一接口层
- MySQL、Redis、对象存储接入
- 用户登录与微信身份体系
- 基础日志、异常处理、鉴权、配置管理
- 文件上传能力

### 产出结果
- 系统具备最基本的前后端联通能力
- 图片、表单、记录类数据可以进入系统
- Java 与 Python 服务可以通过标准接口互调
- 移动端与电脑端具备统一接口接入基础

### 完成标志
- 用户可以完成登录并进入系统
- 前端可以上传图片与表单
- 后端可以完成数据入库
- AI 服务可通过接口返回结构化结果
- 小程序、安卓应用、Web 用户端具备统一鉴权和接口调用能力

### 说明
这一步不能跳过。后续所有饮食识别、OCR、提醒、画像、预警能力都建立在这层基础上。
同时，终端策略在这一阶段就要定清楚：
- 手机端采用微信小程序与安卓应用
- 电脑端采用面向用户的 Web 端
- 管理端继续采用独立后台 Web 系统

## 4.2 路线二：核心闭环开发

### 目标
优先跑通“吃与排”的核心闭环，形成项目第一版最小可用能力。

### 必做内容
- 用户基础档案
- 饮食拍照识别
- 红黄绿风险分级
- 饮食记录落库
- 尿酸记录
- 体重记录
- 发作记录
- 饮水或尿液比色打卡
- 每日提醒与基础趋势图

### 推荐实现顺序
1. 先做用户基础档案
2. 再做记录中心
3. 再接饮食识别服务
4. 再做风险结果展示
5. 最后补提醒和趋势展示

### 核心原因
- 没有档案就无法形成个体化建议
- 没有记录中心就无法沉淀行为数据
- 没有 AI 识别结果就无法体现产品差异化
- 没有提醒和趋势反馈就无法形成留存闭环

### 产出结果
- 用户可以拍照识别饮食风险
- 用户可以记录自身关键指标
- 系统可以给出基础提醒和风险反馈
- 产品具备“用一次就能感知价值”的能力
- 小程序优先承接首版高频功能，安卓应用与 Web 用户端按统一接口逐步补齐

### 完成标志
- 餐盘图片可以返回稳定的风险等级结果
- 用户可以连续记录尿酸、体重、发作情况
- 首页可展示至少一条趋势视图
- 系统可以触发基础提醒消息

### 注意事项
- 首版不要追求精确克重和绝对嘌呤值
- 以“高风险识别 + 定性建议”为主
- 尿液比色可以先做轻交互版本，不必一开始做复杂动画

## 4.3 路线三：可信能力建设

### 目标
在核心闭环稳定后，增强系统对数据的理解能力和用户对平台的信任感。

### 必做内容
- 化验单 OCR 解析
- 关键指标结构化抽取
- 指标趋势图增强
- 指南知识库导入
- RAG 问答能力
- 高风险问答规则兜底
- 用户画像标签体系

### 推荐实现顺序
1. 先做化验单 OCR 与指标标准化
2. 再做知识库整理与问答系统
3. 最后建设画像标签体系与用户诱因总结

### 核心原因
- OCR 能直接补齐真实医疗数据来源
- 知识库问答能增强平台专业可信度
- 标签画像是后续个性化干预与主动预警的前提

### 产出结果
- 用户可以上传化验单并自动解析关键指标
- 用户可以获得基于权威内容的问答反馈
- 系统可以开始总结个人风险因素

### 完成标志
- 化验单关键字段可稳定提取并入库
- 问答结果具备来源约束
- 用户画像至少形成基础标签集

### 注意事项
- 该阶段的问答定位是“健康管理辅助”，不是“替代医生诊断”
- 药物、急性发作、并发症相关问答必须先规则审查再返回

## 4.4 路线四：主动管理能力建设

### 目标
让系统从“被动记录工具”升级为“会主动发现风险并提醒用户的管理系统”。

### 必做内容
- 天气接口接入
- 饮食与天气联合规则引擎
- 用户近期行为风险计算
- 主动预警消息
- 复查提醒
- 药物管理
- 发作复盘报告

### 推荐实现顺序
1. 先接入天气与外部事件数据
2. 再建立规则引擎
3. 再做主动预警通知
4. 再补复查提醒和发作复盘

### 核心原因
- 主动预警必须建立在稳定数据输入之上
- 没有规则引擎，主动提醒就容易失真
- 发作复盘是把“记录”转化为“理解”的关键桥梁

### 产出结果
- 系统可以根据饮食、指标、天气生成风险提示
- 用户开始感知到“平台在主动帮我管理”
- 产品从记录工具升级为半主动管理系统

### 完成标志
- 能生成多因素联合风险提醒
- 能自动触发复查类消息
- 能生成一次完整的发作复盘说明

### 注意事项
- 第一版主动预警优先采用规则模型，不急于上复杂机器学习预测
- 风险等级要保守，不要为了“智能感”输出过强结论

## 4.5 路线五：用户依赖与协同能力建设

### 目标
在主动管理能力具备基础效果后，进一步建立用户关系依赖、数据依赖与家庭协同。

### 必做内容
- 家属绑定
- 亲友监督消息
- 健康喜报与高危提醒
- 个人高危食物总结
- 个人诱因雷达
- 连续记录成长反馈

### 推荐实现顺序
1. 先做家属绑定关系
2. 再做通知模板和授权机制
3. 再做个人诱因总结与成长反馈

### 核心原因
- 家属绑定是高粘性机制，但必须建立在用户信任前提上
- 数据总结越清晰，用户越容易形成“越用越懂我”的感知
- 这一步是平台从“有用”走向“离不开”的关键阶段

### 产出结果
- 用户可以邀请家属共同参与管理
- 系统能反馈个人专属风险模式
- 用户开始对平台产生持续依赖

### 完成标志
- 家属关系可建立、解除、授权控制
- 系统可自动输出个人高危诱因总结
- 系统可展示连续记录带来的画像成长结果

### 注意事项
- 家属监督必须默认由用户主动授权
- 提醒内容要控制强度，避免让产品产生压迫感

## 4.6 路线六：平台生态扩展

### 目标
在主链路和用户依赖稳定后，扩展平台生态与商业承接能力。

### 可做内容
- 第三方设备接入
- 蓝牙尿酸仪数据同步
- 智能水杯联动预留
- 积分体系
- 商品兑换与权益体系
- 运营服务包

### 推荐实现顺序
1. 先做第三方设备数据接入能力
2. 再做积分体系
3. 再做商城或权益体系
4. 最后评估是否自研硬件

### 核心原因
- 硬件和商城都属于重投入模块
- 如果前面没有形成稳定使用和真实留存，这些投入会非常低效
- 第三方接入优先于自研，可以显著降低试错成本

### 产出结果
- 平台具备生态扩展与商业化承接基础
- 用户管理行为与权益激励形成联动
- 系统逐步具备完整平台属性

### 完成标志
- 可以接收并处理外部设备数据
- 用户可通过行为积累积分
- 至少形成一种可验证的权益兑换机制

### 注意事项
- 现阶段不建议过早投入区块链方案
- 优先做普通积分体系即可

## 五、可并行开发的工作流

在顺序不变的前提下，以下工作可以并行推进：

### 5.1 前端与主业务并行
- 小程序页面框架
- 安卓应用页面框架预留
- Web 用户端页面框架预留
- 用户中心
- 记录中心
- 首页趋势展示

### 5.2 Java 与 Python 并行
- Java 先定义结构化接口
- Python 按接口产出识别和解析结果
- 双方通过 mock 数据提前联调

### 5.3 后台与知识库并行
- 管理后台配置页
- 指南资料整理
- OCR 字段标准定义
- 风险规则配置表设计

### 5.4 规则体系与提醒体系并行
- 风险等级规则
- 消息模板
- 订阅消息触发条件
- 用户授权流程

## 六、开发中必须提前控制的风险

### 6.1 医疗风险
- 不直接提供高风险处方建议
- 所有专业建议必须来源可追溯
- 高风险问题必须有降级机制

### 6.2 模型风险
- 饮食识别允许模糊分级，不强求精确数值
- OCR 与问答输出必须可纠错、可复核
- 模型返回结构必须标准化，不能完全自由文本

### 6.3 架构风险
- 首期不堆过多微服务
- 首期不强上复杂多 Agent
- 首期不为了未来扩展而过度设计

### 6.4 产品风险
- 不要一开始就分散到太多功能
- 不要在留存未验证时重投入硬件和商城
- 不要用“未来平台想象”替代“当前用户价值”

## 七、建议的实际执行顺序

如果按最务实的方式推进，推荐按下面的顺序落地：

1. 完成基础工程与数据底座
2. 明确手机端与电脑端边界，完成小程序、安卓应用、Web 用户端统一接入方案
3. 完成用户中心、档案中心、记录中心
4. 接入饮食识别并形成红黄绿反馈
5. 完成提醒与趋势展示，跑通第一版闭环
6. 接入化验单 OCR 与指标解析
7. 建设知识库问答与安全规则
8. 建设画像标签、诱因总结与药物管理
9. 建设天气联合预警、复查提醒与发作复盘
10. 建设家属绑定与亲友监督
11. 评估并扩展设备、积分与平台生态

## 八、最终结论

这份开发路线的核心不是把所有功能线性铺开，而是明确一个判断标准：
每往前一步，都必须让系统离“主动管理平台”更近，同时又不让项目在前期被高复杂度和高风险功能拖住。

因此，后续开发应坚持以下主线：
- 先做基础底座
- 再做核心闭环
- 再做可信能力
- 再做主动管理
- 最后做平台生态

只有这样，项目才能既保住终局方向，又提高一次性做成的概率。
