# 痛风小助手优化跟进

## 一、文档目的
本文档基于以下三部分信息整理：

- 原始项目规划书
- 已形成的《技术架构.md》与《开发路线.md》
- 产品设计同事提交的市场与产品评估清单

本文档的目标不是推翻现有方向，而是在“主动管理平台”总目标不变的前提下，结合当前开发进度，对产品定位、MVP 范围、功能优先级、风险边界和后续执行策略进行一次收缩与校正。

## 二、先给结论

### 2.1 方向结论
项目可以继续推进，但必须停止“继续横向扩功能”的惯性，回到“先验证 MVP 是否成立”的节奏。

### 2.2 产品结论
市场需求真实存在，但不是“功能越多越容易成功”的项目。真正决定成败的不是功能总量，而是以下三点：

- 用户是否愿意持续记录
- 用户是否信任系统给出的风险判断
- 产品是否能够在合规边界内建立长期留存

### 2.3 执行结论
当前仓库的实际开发进度已经明显快于产品验证节奏。后端和 Web 前端已经进入：

- 主动管理
- 家属协同
- 设备接入
- 成长激励

这些能力本身没有错，但在 MVP 留存与商业化尚未验证之前，不应继续作为当前阶段的主要投入方向。后续应执行“功能冻结 + MVP 回收 + 验证优先”的策略。

## 三、结合评估清单后的新判断

### 3.1 市场判断
保留原结论：有真实需求，属于中高需求等级，但不是天然爆款需求。

本项目适合切入的不是泛健康市场，而是痛点更强、管理意愿更高、已经接近长期管理状态的用户。

### 3.2 用户判断
首要目标用户应收缩为：

- 20至 55 岁男性
- 近一年有反复痛风发作经历
- 已有尿酸仪、体检记录或复查习惯
- 对饮食控制有明确焦虑
- 有配偶或家属监督场景

暂不作为首要目标人群：

- 泛减脂和泛代谢综合征用户
- 仅对健康饮食有兴趣但无痛风风险的普通人群
- 需要深度诊疗闭环的医院端场景

### 3.3 产品判断
现阶段用户不会因为“多智能体、长期记忆、成长积分、设备生态、主动平台愿景”留下来。

用户更可能因为以下问题留下来：

- 这顿饭能不能吃
- 为什么这次尿酸又高了
- 我的化验单到底是什么意思
- 最近是不是该复查了
- 家里人能不能知道我最近控制得怎么样

所以当前产品表达也应同步收缩。

## 四、新版产品定位

### 4.1 北极星不变
项目长期目标仍然是“痛风主动管理平台”。

### 4.2 当前对外定位调整
建议把现阶段产品定位从：

“AI 痛风全病程主动管理平台”

调整为：

“痛风饮食与尿酸风险管理助手”

或者更具体一点：

“拍照识别饮食风险，自动生成尿酸管理报告的痛风助手”

### 4.3 调整原因
- 这个表达更贴近用户能立即理解的价值
- 更适合冷启动获客
- 更符合当前 MVP 应该验证的能力边界
- 能有效降低过度承诺带来的信任和合规风险

## 五、结合当前开发进度的现状复盘

## 5.1 当前已完成或已具备的能力

根据当前仓库结构、README 和已存在代码，项目并不是从 0 开始，而是已经具备较完整的后端与 Web 联调基础。

### 已具备的核心闭环能力
- 模拟登录与基础鉴权
- 用户档案管理
- 文件上传
- 餐盘识别闭环
- 尿酸记录
- 体重记录
- 发作记录
- 饮水/尿液打卡
- 首页总览
- 趋势图数据
- 提醒列表
- 时间线记录

### 已具备的第二层能力
- 化验单 OCR
- 痛风知识问答
- 用户画像总结
- 用药管理

### 已进入甚至超前开发的能力
- 天气联合主动管理
- 家属协同与亲友监督
- 设备接入
- 成长积分和挑战体系
- 多指标设备接入

### 已存在的前端页面形态
当前 `frontend-web` 已有页面包括：

- `OverviewPage`
- `RecordsPage`
- `AnalysisPage`
- `AssistantPage`
- `ProactivePage`
- `FamilyPage`
- `DevicesPage`
- `GrowthPage`

这说明系统已经不是“仅有 P0 原型”，而是已经发展成较完整的平台雏形。

## 5.2 当前最大问题
问题不在于“做得慢”，而在于“做得太快、太广”。

现阶段的主要矛盾是：

- 产品验证还没完成
- 但平台能力已经提前铺开

这会导致三个风险：

- 研发资源被分散到非关键闭环
- 用户价值没有被充分验证，系统却已经变重
- 后续很难判断到底是需求不成立，还是产品过早复杂化

## 六、新版范围策略

### 6.1 策略总原则
从现在开始，执行以下策略：

- 停止新增非 MVP 功能
- 优先打磨已经存在的核心闭环
- 让“平台能力”退到后台预留，而不是继续前台扩张

### 6.2 新版范围划分

#### A 类：当前必须继续做强的能力
- 饮食拍照识别
- 红黄绿风险提示
- 尿酸记录与趋势
- 基础原因分析
- 化验单 OCR
- 家属轻量提醒

#### B 类：可以保留，但不再继续扩展的能力
- 天气联合风险提醒
- 用药管理
- 用户画像总结
- Web 用户端趋势与报告查看

原则：
- 已做的可以保留
- 但当前版本不再继续往更复杂方向扩

#### C 类：立即冻结的能力
- 积分商城
- 复杂成长体系
- 自研硬件相关设计
- 区块链表达
- 复杂多 Agent 编排
- 精确发作概率预测
- 精确药物剂量建议
- 3D 疼痛热力图

## 七、新版 MVP 定义

## 7.1 MVP 目标
验证产品是不是一个值得继续加重的平台，而不是先假设它一定会成为平台。

### MVP 只验证五件事
1. 用户是否愿意拍照记录饮食
2. 用户是否愿意持续记录尿酸
3. 用户是否信任风险提示和趋势分析
4. 用户是否认可化验单 OCR 的专业感
5. 家属提醒是否能提升留存和复用

## 7.2 MVP 功能范围

### 必须保留的 5 个核心功能

#### 1. 饮食拍照识别
输出方式：

- 高风险
- 中风险
- 低风险

要求：
- 不追求精确克重
- 不追求绝对嘌呤值
- 重点解决“能不能吃、要不要避开”的判断

#### 2. 尿酸记录与趋势
能力包括：

- 手动录入
- 化验单录入
- 趋势展示
- 超标提示

#### 3. 超标原因分析
现阶段优先采用规则驱动，不追求复杂 AI 预测。

重点关联：

- 饮食记录
- 饮酒
- 饮水
- 熬夜
- 发作记录

#### 4. 化验单 OCR 解读
重点提取：

- 尿酸
- 肌酐
- 血糖
- 血脂
- 其他核心代谢指标

目标不是替代医生，而是让用户“看懂重点”和“知道是否需要复查”。

#### 5. 家属轻量提醒
只做两类内容：

- 健康摘要
- 风险提醒

不做过强监督，不做隐私越界，不做复杂社交传播。

## 八、当前版本不建议继续推进的功能

以下功能即使代码已有雏形，也不应作为当前版本的继续投入重点：

- 成长积分体系
- 周挑战和勋章体系
- 奖励兑换
- 多指标设备深度扩张
- 智能水杯或自研硬件方向
- 复杂知识代理与长期记忆
- SOS 精确用药指导
- 发作概率数字化表达

处理原则：

- 保留代码基础
- 不删除已有成果
- 但在当前版本里不继续前台强化
- 产品展示与研发资源都回到 MVP 主链路

## 九、合规与信任边界再确认

### 9.1 必须坚持的边界
本产品属于健康管理工具，不是诊疗替代系统。

### 9.2 当前所有文案与交互必须遵守
- 不将系统描述为 AI 医生
- 不直接给出处方级建议
- 不直接输出药物剂量
- 不在信息不足时输出强结论
- 不宣传未经验证的“发作概率”

### 9.3 推荐的表达方式
把以下类型表达替换掉：

- “今晚发作概率 60%”
- “立即服用某药某剂量”

替换为：

- “近期存在多重风险因素叠加，建议加强饮水、注意保暖、避免高嘌呤饮食，并根据既往医嘱处理”
- “若疼痛剧烈、持续加重或伴随异常症状，请及时线下就医”

## 十、终端策略同步优化

当前终端方向不变，但要重新强调每个端的职责，避免所有端都做成“大而全”。

### 10.1 微信小程序

### 10.2 安卓应用

### 10.3 Web 用户端

## 十一、对现有开发路线的校正

## 11.1 路线不推翻，但增加“冻结线”
现有《开发路线.md》的主线仍然成立：

- 先做核心闭环
- 再做可信能力
- 再做主动管理
- 最后做平台生态

但结合当前代码进度，必须新增一条执行规则：

### 当前开发冻结线
在以下指标未验证之前，不再继续扩展平台生态能力：

- 7 日留存
- 30 日留存
- 每周拍照频次
- 家属提醒使用率
- OCR 上传使用率

## 11.2 当前建议的实际工作顺序调整
从现在开始，推荐按下面顺序处理：

1. 盘点现有功能，划分 MVP 保留项与冻结项
2. 调整对外产品定位与文案表达
3. 集中打磨饮食识别、记录、趋势、OCR、家属摘要五条主链路
4. 收敛 Web 前端页面，只保留当前验证所需入口
5. 暂停积分、挑战、复杂设备生态的前台投入
6. 建立灰度验证与数据埋点
7. 用真实用户数据判断是否进入下一阶段

## 十二、建议保留页与建议降级页

### 12.1 当前建议保留的前台页面
- `OverviewPage`
- `RecordsPage`
- `AnalysisPage`
- `AssistantPage`

说明：
- `OverviewPage` 承接总览和提醒
- `RecordsPage` 承接记录闭环
- `AnalysisPage` 承接趋势和原因分析
- `AssistantPage` 承接化验单和知识辅助

### 12.2 当前建议降级处理的页面
- `ProactivePage`
- `FamilyPage`
- `DevicesPage`
- `GrowthPage`

降级含义不是删除，而是：

- 入口隐藏
- 只保留最小可测能力
- 不作为当前对外主卖点

其中：
- `FamilyPage` 可以保留最基础摘要与授权绑定
- `ProactivePage` 只保留保守型风险提示
- `DevicesPage` 与 `GrowthPage` 暂不作为 MVP 重点页面

## 十三、MVP 验证指标

上线前后，建议围绕以下指标判断 MVP 是否成立。

### 13.1 关键指标
- 落地页留资率：`>= 8%`
- 注册后首日完成记录率：`>= 40%`
- 7 日留存：`>= 25%`
- 30 日留存：`>= 10%`
- 每周饮食拍照次数：`>= 3 次/人`
- OCR 使用率：持续提升，并进入稳定使用
- 家属绑定率：用于验证监督机制价值
- 付费意愿：至少有一部分用户愿为报告版或家庭版付费

### 13.2 判断方式
如果拍照、记录、OCR、家属提醒四条链路中有两条以上始终跑不通，则说明问题不只是功能未完善，而是当前产品形态尚未打中核心价值。

## 十四、各团队接下来的动作建议

## 14.1 产品侧
- 重写对外介绍与版本说明
- 明确当前版只主推五个核心能力
- 输出更具体的用户旅程和冷启动话术

## 14.2 Java 后端
- 继续稳住核心记录、提醒、趋势、OCR 入库链路
- 对已完成的家属、设备、成长能力增加开关化控制
- 优先保证现有主链路稳定，不再继续横向新增复杂业务

## 14.3 Python AI 服务
- 优先提升 OCR 稳定性和饮食识别结果可解释性
- 问答系统继续维持知识库约束
- 不扩展高风险医疗建议能力

## 14.4 前端
- 收敛导航结构
- 优先优化记录、拍照、趋势、OCR 上传、家属摘要的操作路径
- 减少会分散注意力的入口

## 14.5 运营与验证
- 用病友群、微信生态、小红书等渠道做小样本验证
- 先测真实使用与复用，不急于做大规模推广
- 建立用户访谈和行为数据回收机制

## 十五、最终跟进结论

这个项目不需要换方向，但需要换节奏。

正确做法不是继续把“主动管理平台”的所有想象一次性做完，而是先把已经做出来的系统收回来，压缩成一个用户愿意反复使用的 MVP。

因此，本轮优化跟进后的统一策略应为：

- 北极星保持“主动管理平台”不变
- 当前定位收缩为“痛风饮食与尿酸风险管理助手”
- 当前版本只聚焦五个核心功能
- 当前开发进入“冻结扩张、强化闭环、验证留存”的阶段
- 等真实指标成立后，再决定是否恢复平台生态扩展

这才是最符合当前项目进度、市场判断和合规现实的下一步方案。
